## 1. Executive Summary

A **production-grade, multi-tenant CRM + Business Intelligence platform for network-marketing
organizations**, designed end-to-end on a locked stack:

- **Backend / system of record:** Supabase — **Postgres 15** with **Row-Level Security as the
  primary isolation boundary**, Supabase Auth (GoTrue), Edge Functions (Deno), Realtime, and
  `pg_cron`. The design is deliberately **database-centric**: tenant isolation, genealogy
  invariants, and transactional domain logic live *inside* Postgres so no client or Edge layer can
  bypass them.
- **Frontend:** **Next.js 14 App Router** + TypeScript (strict) + Tailwind + shadcn/ui + Recharts,
  Italian-first via `next-intl` (i18n-ready, locale-segmented routes), React Flow + d3-hierarchy
  for the binary genealogy canvas, `@tanstack/react-query` + Supabase Realtime for live state.
- **Multi-tenancy:** every tenant table carries `org_id uuid`; isolation is enforced by RLS keyed
  on the `org_id` JWT claim. **No cross-org access is possible at the database layer** (`FORCE ROW
  LEVEL SECURITY` on every tenant table).
- **Binary genealogy (the signature model):** a **true binary placement tree** —
  `marketers.parent_id` + `marketers.leg ('LEFT'|'RIGHT')`, one LEFT + one RIGHT child per parent
  enforced by a partial unique index. A **separate `sponsor_id`** records the actual recruiter
  (decoupled from placement to support **spillover**). Subtree/branch queries are made O(index) by a
  **closure table** (`marketer_tree_closure` with a `branch_leg` tag) **plus** an **ltree
  materialized `path`** — closure for math/visibility, ltree for drawing. The closure table is the
  **single visibility primitive** for RLS and the **single aggregation primitive** for analytics.
- **Marketer profile ≠ user account (mandatory separation):** a `marketers` profile exists with no
  login (pre-registration); a `memberships` row later attaches an `auth.users` login to that
  **existing** profile. Activation ("Activate CRM Access") never recreates or duplicates the
  profile — `id`, genealogy, history, contacts, documents, stats all survive.
- **Authorization (four orthogonal axes):** org role (`memberships.role`), rank
  (`marketers.rank`), per-account permission flags (`memberships.permissions`), and genealogy
  position (closure subtree). A user sees **own + entire downline subtree**, never uplines or
  parallel branches; admins/owners see the whole org. Rank gates **CRM eligibility**
  (Executive = no CRM unless an admin override sets `permissions.crm_access`).
- **Analytics / BI engine:** one atomic fact grain (`daily_marketer_metrics`, per marketer × day),
  with subtree/branch aggregation **computed on read** via the closure join; two materialized views
  (`mv_funnel_totals`, `mv_stage_conversion`) for funnel/conversion; immutable snapshot tables for
  `monthly_reports`, `leaderboard_snapshots`, `bottleneck_findings`. Refresh is two-tier
  (trigger-driven dirty set + 15-min MV refresh for hot; nightly `pg_cron` for cold), all
  idempotent. Global / Left Branch / Right Branch is a single `branch_leg` predicate everywhere.
- **Reporting:** five canonical report types + automatic monthly/quarterly reports (MoM/QoQ diff +
  %, historical evolution) generated by `pg_cron`, rendered to PDF/XLSX/CSV in a Deno Edge Function
  with **assembly in SQL (RLS-scoped), rendering in Deno** — compute once, render many.

## 5. Cross-document strengths worth preserving

- **One visibility primitive** (`can_see_marketer()` over the closure table) is used identically by
  RLS, search, analytics MV wrappers, report assembly, and the API authorization gates — change it
  once, change it everywhere.
- **Compute-once-render-many**: dashboards, reports, and exports all read the same closure-joined
  facts / snapshots; the export layer owns no metric math, so a PDF and the screen never disagree.
- **Defense-in-depth that does not rely on any single layer**: RLS alone prevents cross-org and
  cross-branch leakage even if the app or an Edge Function is compromised.

## 7. Recommended Build Sequence

Implement in dependency order once the §6 decisions (especially #2, #13, C-1, C-5) are signed off.

1. **Foundation migration** — extensions (`pgcrypto`, `ltree`, `btree_gist`, `pg_trgm`, `pg_cron`),
   all enum types, Group 1–2 tables (`organizations`, `memberships`, `ranks_meta` seeded,
   `marketers`, `marketer_tree_closure`), the binary-tree integrity constraints, and the
   closure/ltree maintenance triggers (14 §2). *This is the spine — nothing else works without it.*
2. **Auth + JWT hook + the visibility primitive** — Supabase Auth config (public signup disabled),
   the canonical access-token hook (decision #2), and `can_see_marketer()` / `is_org_admin()` /
   claim accessors. Enable `FORCE ROW LEVEL SECURITY` and the `marketers`/closure RLS policies.
3. **Account lifecycle** — `account_invitations`, the eligibility guard trigger, and the
   `create-invitation` / `activate-account` Edge Functions + `accept_invitation` RPC
   (profile-preserving activation). Then `place_marketer`/`move_marketer` per the reconciled C-2
   contract, with `rank_history` + the structural-column guard trigger.
4. **Core CRM data + RLS** — `contacts`, `centos_list_entries`, `seven_whys`, `prospects`,
   `prospect_journey_events` (+ `change_prospect_stage()`), `calls`, `internal_documents` /
   `document_versions`, each with its per-table RLS policies and the indexes from `01` (+ the `13`
   NEW indexes via `CONCURRENTLY` on a populated DB).
5. **App shell + navigation** — Next.js `(crm)` shell, the CRM gate + layout guards, the reconciled
   route map (C-5), `next-intl` catalogs, design tokens, and the scope/branch provider.
6. **Genealogy UI** — the React Flow tree (lazy expand, node payload contract 14 §7, search,
   branch tabs, admin drag-to-move).
7. **CRM screens** — contacts (search/filter/tags/bulk/follow-up), Centos, Sette Perché, the
   6-stage prospect funnel board, call logging — the daily-driver surfaces.
8. **Analytics fact layer** — `daily_marketer_metrics`, the dirty-set queue (single canonical name,
   C-4) + triggers + `drain` job, the two MVs + 15-min `pg_cron` refresh, and the `*_for_scope`
   secured analytics functions/views.
9. **Analytics & dashboards UI** — rank-adaptive dashboard, performance/conversion/team/branch
   pages, the Global/Left/Right switcher wired to the closure predicate.
10. **Leaderboards + bottleneck engine** — `leaderboard_snapshots` + refresh, the bottleneck rule
    engine + `bottleneck_findings` + notifications, and `notifications` Realtime.
11. **Reporting + export** — `monthly_reports` generation (`pg_cron` dispatcher), the
    assembly→render pipeline, `report_export_jobs` + `reports` bucket, PDF/XLSX/CSV, the CEO
    dashboard, and `audit_log` wiring across all sensitive actions.
12. **Hardening & scale** — MFA step-up, full audit coverage + immutability, rate limits, then the
    scale levers from `12`/`13` (read replica routing, partitioning, per-org incremental refresh)
    introduced as the capacity tiers demand, not up front.

> Cross-cutting throughout: write every sensitive action to `audit_log`, keep every `pg_cron` job
> idempotent, and add the `13` NEW indexes as their queries come online.
